# CLAUDE.md — slimsplaydy

Notes for future sessions. Workspace-wide conventions live in `../CLAUDE.md`; this file holds only
what's specific to **slimsplaydy**. Keep it thin. (Name is lowercase `slimsplaydy` in repo/folder
and filenames; the meta `name:` / silk reads "SlimSplaydy".)

## What this is

- The original **Slim Splaydy** — a **34-key** low-profile splay split: a bit of splay, quite a bit
  of finger-column stagger, two-key thumb fan per hand.
- One **nice!nano (nRF52840)** per half, mounted **SMD upside-down** in a board cutout. Switches are
  **Kailh PG1316S** ultra-low-profile. Everything is SMD, front side only.
- Lineage: began as the PG1316S version of `../splave-ferris`, evolved well past it (Choc variant
  removed; collapsed to a single version with `config.yaml` at the repo root — **no variant
  subfolders**). It is the **parent of `../xiphos`**, which forked from this repo 2026-06-03 and then
  diverged (36 keys, single combined board). Copies xiphos forked from live in
  `../xiphos/_old_slimsplaydy/` — the working repo is **here**.

## Status — routed + PCBA-tested. PRESERVE ROUTING.

`slimsplaydy_left.kicad_pcb` and `_right` are **fully routed** and have a JLCPCB **PCBA test order**
behind them (`production/`, `Production_JLCPCBA/`). **Do not regenerate from ergogen** — a regen wipes
routing and forces a full re-route. Only regenerate if a real functional defect demands it. Edits to
`config.yaml` are for keeping the source-of-truth honest, **not** for re-emitting the PCB.

## Design concept (shared with `../xiphos`)

Ultra-thin **travel** split that clamshells shut. Thesis: PG1316S are so low that when pressed they
drop below the nice!nano's USB-C receptacle, so the **USB-C is the thickness floor** — "the thinnest a
board can be while keeping an onboard USB-C port." Enablers:

- **nice!nano SMD upside-down** in a board cutout (`mcu_slot_*` outline) so the USB-C sits flush with
  the PCB bottom and the nano's back faces up.
- **PCB cut away under the battery** (`batt_slot_*` outline); the case holds the cell, which stays
  under the USB-C height.
- **Blind slot milled in the back** under the `magsafe` silk ring — **route no traces across it** —
  holding a **steel ring** flush with the board bottom. The ring is for **mounting** to Hunter's
  existing MagSafe accessories (leg straps, desk clamps, tent stands), **not** for clasping the halves.
  Use ferromagnetic steel (low-carbon / 400-series, not 304), sized to the MagSafe array.
- **Case** (`Case_Models/*.step`): open-bottom ring + battery holder + clasp magnets, rim only as tall
  as a depressed key. Halves fully separate (no hinge). Closed stack is the design's reason to exist.

## Blind slot — fab detail (not in the KiCad files)

- Geometry comes from the `magsafe_silkscreen` footprint zone on **B.SilkS**: inner Ø44.8 / outer
  Ø55.2 mm (annular band, radius 22.4–27.6 mm) centered on MAG1. That zone is a **visual cue only**
  (`tracks allowed` — KiCad will **not** stop you routing across it; avoid it by hand).
- **Depth is specified in the JLC order, NOT the design files.** Ordered **0.8 mm** deep (≈ halfway
  through a 1.6 mm board). At that depth **F.Cu is untouched** — only **B.Cu and vias** in the band
  matter. Back copper was hand-routed clear of the ring; keep it that way on any re-route.

## Board structure (kept as-is — do NOT unify)

- **Two separate routed files:** `slimsplaydy_left.kicad_pcb` + `slimsplaydy_right.kicad_pcb`, each its
  own `.kicad_pro` / `.kicad_sch`. Nets are **non-namespaced** (`finger_*`/`thumb_*` left,
  `mirror_*` right; both use plain `GND`/`RAW`/`BAT_P`/`RST`).
- `slimsplaydy_both*.kicad_pcb` are **hand-merged in KiCad** (both halves on one sheet, right nets
  `R_`-prefixed) — used for the combined production order (`JLC_Positions.csv` has CONN1 + R_CONN1).
  They were assembled by hand, **not** generated by ergogen.
- Unlike `../xiphos` (single combined ergogen board, `L_`/`R_` nets), slimsplaydy was **left split**
  on purpose: it's routed + PCBA-tested, and unifying would force a re-route. **Only unify if Hunter
  asks.**

## Refs / nets (verified against the routed PCBs)

- Per half: `MCU1`, `CONN1`, `PWR1`, `RST1`, `MAG1` (silk only), `S1–S17` keys, `MH1–MH15` mounting
  holes (NPTH, no net). Schematic carries the electrical parts; `MAG1` + `MH*` are intentionally
  PCB-only. Schematic ↔ PCB parity is clean.
- Net roles: `CONN1` 1=GND 2=BAT_P · `PWR1` RAW↔BAT_P · `RST1` GND↔RST · keys `<name>`↔GND.

## Production (JLCPCB PCBA)

Fab + assembly at JLCPCB. LCSC: battery connector **`C505023`**; power switch **`C2911519`**; reset
switch **`C79174`**; PG1316S **consigned** **`C9900170245`** (Hunter's reel). **nice!nano = DNP /
hand-soldered** (not in JLC's library). The `production/*.csv` exports have a blank `LCSC Part #`
column — the authoritative order BOM with part numbers is `Production_JLCPCBA/JLCPCB_BOM.xls`.

## Battery connector — the 1.0 → 1.20 mm swap (read before touching `battcon`)

The routed boards use the **regular Molex Pico-EZmate, 1.20 mm pitch**
(`Molex_Pico-EZmate_78171-0002_1x02-1MP_P1.20mm_Vertical`, LCSC `C505023`). The first build used the
**1.0 mm EZmate *Plus*** by accident; the Plus turned out uncommon and **unavailable for PCBA**, so the
part was **hand-swapped** to the 1.20 mm in the PCB and schematic. `config.yaml` was **rectified to
match (2026-06-04, no regen):** both `battcon` blocks now use the ceoloide footprint
`battery_connector_molex_pico_ezmate_1x02` with `BAT_P: BAT_P` / `BAT_N: GND` (pad 1=GND, 2=BAT_P —
matches routed CONN1). The old `footprints/conn_molex_pico_ezmate_plus_1x02.js` is now **unused** (kept,
not deleted).

## Symbols, the nice!nano, and ERC/DRC (2026-06-05 — read before touching the MCU symbol)

- **Symbol library is its own repo now:** `huntercook` lives in **`KiCad_Symbols`**
  (`https://github.com/jusdisgi/KiCad_Symbols`), file **`huntercook.kicad_sym`** (typo `huntercooh`
  killed). Reference it on every machine via KiCad path var **`${HLC_SYMBOLS}`** → the repo dir
  (Preferences → Configure Paths), library entry `${HLC_SYMBOLS}/huntercook.kicad_sym`, nickname
  **`huntercook`**. Schematic `lib_id`s are `huntercook:nice_nano_raw21` / `huntercook:PG1316S`.
- **The library symbols are CUSTOMIZED STARTING POINTS, not the source of truth.** The real
  board-specific symbol (e.g. the upside-down nano with the three middle pads dropped) was tailored
  **in the schematic**. **Do NOT blind-run "Update Symbols from Library"** — it overwrites those
  customizations: it re-added phantom pins and **reset MCU1's Footprint field** to a generic 28-pad
  footprint, which is what spawned the "changing footprint" + "missing pads 25-29" mess this session.
  If you must update, **uncheck the reset-fields options** and re-confirm MCU1's Footprint field after.
- **nice!nano GND pins** are now **`power_input`** (were `power_output`, which caused
  "power output + power output connected" ERC errors when the three GND pins tied together). A single
  **PWR_FLAG on GND** now drives the net. Fixed in the library symbol, so xiphos/lightfury inherit it
  on their next Update-Symbols.
- **MCU1 footprint is `ceoloide:mcu_nice_nano`** (29-pad, embedded as a local override; the ergogen
  module `mcu_nice_nano_smd.js` emits that name). The symbol's Footprint field must read exactly
  `ceoloide:mcu_nice_nano` or F8 tries to swap it. `ceoloide` is **not** in the fp-lib-table → benign
  `lib_footprint_issues` warnings on MCU1/PWR1/RST1/all MH. The board's **embedded** copy is what's
  fabbed — **never "Update Footprints from Library"** on these routed boards.
- **LCSC fields**: populated in both schematics **and** pushed to the PCB footprints (both halves).
- **ERC clean. DRC = 0 errors.** The ~30 remaining DRC items are all **warnings, by-design**:
  `silk_edge_clearance` + `silk_overlap` on MCU1 (silk crosses the nano cutout / its own labels) and
  the `ceoloide` `lib_footprint_issues`. **Exclude** them (same as LightFury's RE1) — don't "fix."

## Panelization (KiKit, JLCPCB) — scripted pipeline (ported from LightFury 2026-06-19)

The old **Option B** plan (hand-`_both` files / KiCad *Append Board* / `kikit fab jlcpcb --assembly`)
is **superseded** — `kikit fab --assembly` needs a schematic the panel's duplicate `_2` refs can't
match. Use the **scripted** pipeline in **`panel/README.md`** instead (modeled on LightFury, which
debugged this end-to-end). On Hunter's machine (needs `pcbnew`; `kikit.exe` in
`...KiCad\9.0\3rdparty\Python311\Scripts`, add to PATH for the session). Per-half outline
≈ **122.5 × 97.3 mm**; side-by-side nest ≈ 250 × 110 mm panel.

1. `python panel/merge_both.py` → `slimsplaydy_both.kicad_pcb`. Side-by-side nest, **no rotation/tilt**
   (every part is already on a whole degree — slimsplaydy needs neither LightFury's 180° flip nor its
   integer `TILT`). Right refs → `*_2`, nets → `L_`/`R_`. Build artifact — don't hand-edit; fix the
   half and re-run.
2. `kikit panelize -p panel/slimsplaydy_panel.json …` — **fixed** tabs + full **frame** + mouse-bites
   (preset switched off `spacing` tabs, which error on the splayed outlines).
3. **Fabrication Toolkit** on `panel/slimsplaydy_panel.kicad_pcb` — **GERBERS only** (not `kikit fab`,
   not its CPL/BOM).
4. `python panel/gen_cpl.py panel/slimsplaydy_panel.kicad_pcb panel/cpl_build` — writes
   `positions.csv` + `bom.csv` (position = footprint origin; pure-Python, no pcbnew). **No RESW
   injection** (slimsplaydy has no encoder — that's a LightFury-only step). Then
   `python panel/render_cpl.py …` and eyeball `verify_*.png` before uploading.

Places 20 parts/half: 17× PG1316S (consigned `C9900170245`) + CONN1 (`C505023`) + PWR1 (`C2911519`)
+ RST1 (`C79174`). Excludes MCU1 (DNP), MH1–15 (NPTH), MAG1 (silk). Scripts: `merge_both.py`,
`gen_cpl.py`, `render_cpl.py`, `kicad_panel.py` (the last two copied verbatim from LightFury).
